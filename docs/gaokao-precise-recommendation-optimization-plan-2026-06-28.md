# 四川志愿填报助手精准推荐优化方案

日期：2026-06-28

## 一句话结论

当前志愿填报助手已经具备“四川考生信息采集、历史位次匹配、冲稳保候选、报告生成和分享”的基础能力，但过去的推荐主要停留在“学校 + 院校专业组”层面。要让用户真正感觉专业、可信、能用，必须把你核实过的四川 2026 官方招生计划数据接进推荐链路，让结果同时给出：

- 推荐学校
- 推荐院校专业组
- 组内建议专业
- 组内不确定性和调剂风险
- 学费、计划数、选科、位次、专业实力等核对依据

这次新增的官方数据表和导入脚本，就是把产品从“院校专业组初筛”推进到“学校 + 专业建议”的第一步。

## 当前产品审查

### 已经专业的部分

- 能基于四川考生的科类、分数、位次、专业偏好、城市偏好、民办/中外/调剂接受度生成报告。
- 推荐结果按“冲一冲、稳一稳、保一保”组织，符合志愿填报用户的理解习惯。
- 不在四川历史数据缺失时编造学校名单，风险口径比较稳。
- 报告里有免责声明、下一步核对清单和风险提醒。
- 已经接入张雪峰 agent 的顾问思路或接口预留，但没有完全依赖它。

### 不够专业的部分

- 原推荐主要来自历史录取记录，很多条目只能显示到“院校专业组”，用户很难判断自己到底适合组内哪个专业。
- 专业偏好更多影响候选排序，还没有充分利用 2026 官方招生计划里的专业名称、计划人数、选科要求、专业备注、专业实力等字段。
- 对“专业组干净度、组内专业数、转专业政策、学费、招生章程”等关键填报风险没有形成稳定的结构化输出。
- 张雪峰 agent 原始数据库不覆盖四川，不能直接作为四川志愿推荐主库；它更适合作为解释话术、顾问风格、问诊流程的补充层。

## 张雪峰 agent 的定位

### 它能不能推荐专业

原张雪峰 agent 的设计可以推荐具体专业，但前提是它的数据库里有对应省份、年份、学校、专业的明细。它的开源数据更偏部分省份样例，不适合直接拿来做四川 2026 的正式推荐主库。

### 在本产品里的正确用法

- 不建议直接把张雪峰 agent 当四川主推荐引擎。
- 建议把它作为“顾问表达层”和“问诊策略层”：
  - 帮用户澄清专业偏好、就业偏好、地域偏好、读研偏好。
  - 解释为什么某个学校适合冲、稳或保。
  - 解释某个专业组为什么需要谨慎。
  - 把数据库结果翻译成用户听得懂的建议。
- 正式学校和专业名单必须以本地官方数据表为准。

## 推荐引擎目标

用户端最终应该拿到四层结果：

### 第一层：学校推荐

回答“这个分数和位次可以重点看哪些学校”。

核心依据：

- 四川历史投档/录取数据
- 用户分数和位次
- 科类和选科
- 城市偏好
- 民办、中外合作、预算接受度
- 冲稳保风险偏好

### 第二层：院校专业组推荐

回答“这个学校里的哪个专业组值得看”。

核心依据：

- 院校专业组代码
- 专业组历史最低位次
- 专业组计划人数
- 专业组内专业数量
- 专业组干净度
- 是否包含用户排斥专业

### 第三层：组内专业推荐

回答“这个专业组里，优先建议看哪几个专业”。

核心依据：

- 专业名称和专业类是否匹配用户偏好
- 2026 预估位次
- 2025/2024/2023 专业最低位次
- 计划人数
- 学费和学制
- 选科要求
- 专业评级、软科排名、学科评估、国家级/省级一流等信息

### 第四层：风险解释

回答“为什么推荐，以及哪里需要谨慎”。

必须提示：

- 专业位次是否比用户当前位次更靠前
- 专业组内是否专业过多、过杂
- 是否有新增专业导致历史位次不足
- 是否有高学费、中外合作、民办、特殊培养地点
- 是否需要核对转专业政策和招生章程

## 当前已实现的最小闭环

### 数据层

新增 `gaokao_major_plans` 表，承接四川 2026 官方招生计划专业明细。

### 导入层

先从 Excel 提取为 NDJSON，再导入数据库。这样做的好处是：

- Excel 只在本机处理一次。
- 服务端部署时只需要导入 NDJSON。
- 导入过程可重复执行，可用 `--replace` 覆盖同一批官方数据。
- Markdown 审计文件保留字段和样例，方便人工复核。

### 推荐层

保留原来的冲稳保候选逻辑，不直接推翻已经能用的学校推荐。

新增逻辑：

- 对每条候选学校/专业组，尝试匹配 `gaokao_major_plans`。
- 如果候选里有“专业组101”这样的文本，就按学校、科类、专业组代码查 2026 官方计划。
- 如果候选已经是具体专业，就按学校、科类、专业名称查对应计划。
- 每条候选最多展示 3 个组内建议专业。

### 页面层

在“冲稳保候选”卡片下新增“组内建议专业”：

- 专业名称
- 计划人数
- 2026 预估位次或 2025 专业位次
- 学费和学制
- 专业备注
- 推荐理由
- 风险提示

## 检索速度和 token 成本控制

### 核心原则

不要让 AI 直接读取 Excel、Markdown 或 5 万多行明细。正确流程是：

1. 数据库负责检索和排序。
2. 程序只取少量候选结果。
3. AI 只看压缩后的摘要，用来解释和追问。
4. 最终学校、专业和位次依据仍来自数据库。

这样既快，也省 token，还能避免 AI 编造数据。

### 当前方案是否能快速检索

可以。`gaokao_major_plans` 已经是结构化数据库表，并且建了常用索引：

- 按年份、省份、科类、批次查候选。
- 按科类和 2026 预估位次查专业。
- 按科类和 2025 专业组位次查参考。
- 按学校代码、专业组代码查组内专业。

用户生成报告时，不需要扫描 51,878 条数据。推荐链路先用历史位次筛出冲稳保候选，再按学校、科类、专业组代码去查该组的少量专业，通常每条候选只取几十条以内，最后只展示 Top 3。

### 用户使用是否费 token

按当前实现，正式推荐本身不费大模型 token：

- 学校推荐：本地数据库计算。
- 专业建议：本地数据库计算。
- 报告结构：本地程序生成。
- 分享页展示：读取已保存结果。

会消耗 token 的主要是“对话式问诊/解释”，并且只有配置了 `GAOKAO_LLM_BASE_URL`、`GAOKAO_LLM_API_KEY`、`GAOKAO_LLM_MODEL` 时才会调用大模型。没有配置时，系统会走本地兜底话术。

### 最省 token 的推荐架构

建议采用“三段式”：

### 第 1 段：确定性检索，不用 AI

本地程序完成：

- 分数转位次
- 冲稳保分档
- 专业组匹配
- 专业偏好过滤
- 学费、选科、民办、中外合作过滤
- 组内专业 Top 3 排序

这一段不调用大模型。

### 第 2 段：小上下文解释，少量使用 AI

如果需要 AI 解释，只传这些压缩字段：

- 考生画像：科类、分数、位次、偏好、排斥项、预算
- 每档候选数量
- 每档最多 3-5 所学校
- 每所学校最多 3 个建议专业
- 每个专业只保留专业名、计划数、位次、学费、风险标签

不要传：

- 全量 Excel
- 全量 NDJSON
- 全部 36 条候选的完整备注
- 招生章程长文本
- 专业组内所有专业的长备注

### 第 3 段：结果落库，重复查看不再用 AI

报告生成后，把结构化 `recommendations` 和 `summary` 保存到数据库。用户后续查看、分享、截图都直接读库，不再次调用 AI。

### token 上限建议

为了稳定控制成本，建议写成硬规则：

- 对话问诊：每轮只问一个关键问题，最多传当前画像和最近一轮用户输入。
- 生成报告：最多传 12 所学校，分布为冲 4、稳 4、保 4。
- 每所学校最多传 3 个建议专业。
- 每个专业备注截断到 80 字以内。
- 招生章程、转专业政策等长文本默认不进 prompt，只在页面提供“需人工核对”提示。

### 更进一步的省 token 方法

- 专业匹配优先用关键词和专业类字段，不急着上向量库。
- 热门专业解释可以做本地模板，例如计算机、临床医学、法学、师范、护理、土木、机械等。
- 同一用户的报告结果落库缓存，修改画像后才重新计算。
- 同一学校专业组的专业建议可以缓存 24 小时。
- AI 只做“解释为什么”，不做“重新推荐什么”。

### 什么时候才需要向量检索

第一版不建议上向量库，MySQL 查询已经够用。只有出现这些需求时再加：

- 用户用自然语言描述职业目标，比如“我想做稳定但别太卷的工作”。
- 用户不知道专业名，需要把兴趣映射到专业类。
- 需要做“专业说明书”“就业方向”“适合人群”的语义匹配。

即使后面加向量，也只向量化小体量的专业词典、职业方向和本地解释模板，不向量化 5 万条招生明细。

## 专业推荐排序规则

第一版排序建议保持透明、可解释，不做黑盒概率。

### 加分项

- 匹配用户专业偏好：高权重
- 匹配用户城市偏好：中权重
- 专业位次接近用户位次：高权重
- 计划人数更多：小幅加分
- 专业评级、学科评估、专业水平较好：中权重
- 专业组干净度更高：小幅加分

### 硬过滤项

- 用户明确排斥的专业方向
- 用户明确排斥的城市
- 用户不接受中外合作，但专业备注含中外合作/合作办学
- 用户不接受民办，但院校性质为民办或独立学院
- 用户设置学费上限，且专业学费超出预算
- 用户选科不满足专业选科要求

### 风险提示项

- 专业参考位次明显比用户当前位次更靠前
- 组内专业数量过多
- 新增专业或缺少历史专业位次
- 只匹配到专业组，没有匹配到具体专业

## 用户端表单还应该补的问题

为了让专业推荐更准，建议后续新增这些轻量问题：

- 更看重学校层次还是专业方向
- 是否准备考研
- 是否倾向体制内、稳定就业、互联网、医疗、教育、金融、制造业等方向
- 是否接受数学/物理/化学强度高的专业
- 是否排斥医学、师范、土木、机械、材料、农学、护理、管理类等方向
- 是否接受省外
- 家庭年度学费预算
- 是否接受专业调剂
- 是否接受大类招生后分流不确定性

这些问题不要一次性做成长问卷，建议放在对话里分轮确认，避免用户流失。

## 页面体验优化建议

### 报告首屏

首屏应该直接告诉用户：

- 你当前更适合“稳档主导 / 冲档可少量 / 保底必须留足”
- 本次筛到多少所学校、多少个专业组
- 是否已使用 2026 官方招生计划数据
- 是否存在数据缺口

### 候选列表

每条候选建议采用固定结构：

- 学校名称
- 专业组名称/代码
- 冲稳保标签
- 历史位次差
- 组内建议专业 Top 3
- 风险提示
- 下一步核对动作

### 风险表达

不要只写“仅供参考”。要写具体：

- “这个专业组内有 12 个专业，如果不接受其中的 X、Y，需要谨慎。”
- “计算机类匹配你的偏好，但 2025 专业位次比你靠前，适合冲，不适合当稳。”
- “该专业计划人数 2 人，波动会比较大。”

## 后续分阶段路线

### 第 1 阶段：官方专业计划入库

目标：让推荐结果从“学校 + 专业组”升级到“学校 + 专业组 + 组内建议专业”。

验收：

- 数据表 `gaokao_major_plans` 有 51,878 条四川 2026 专业计划记录。
- 用户生成报告时，每条能匹配到专业组的候选能展示组内建议专业。
- 对无法匹配的候选，不影响原推荐生成。

### 第 2 阶段：专业组风险评分

目标：不要只推荐“看起来好”的学校，还要识别组内坑。

新增指标：

- 专业组内专业数量
- 专业组干净度
- 用户排斥专业占比
- 高学费/中外/民办风险
- 转专业政策风险

验收：

- 每条专业组都有“推荐、谨慎、需人工核对”三类风险标识。
- 报告能指出“为什么这个学校适合，但这个专业组不一定适合”。

### 第 3 阶段：张雪峰 agent 顾问层增强

目标：让产品更像真人顾问，而不是只吐数据库列表。

做法：

- 数据库先生成结构化推荐。
- 张雪峰 agent 或本地顾问提示词只负责解释、追问和表达。
- 禁止 agent 编造数据库外的学校、专业和录取概率。

验收：

- 输出里每个学校和专业都能追溯到本地数据库。
- 顾问话术能解释取舍，但不替代数据推荐。
- 用户偏好不足时，先追问关键问题，而不是直接下结论。

### 第 4 阶段：人工复核后台

目标：让运营人员能复核和干预重点推荐。

建议能力：

- 搜索学校、专业组、专业
- 查看某考生推荐依据
- 标记不推荐专业组
- 添加人工备注
- 导出给用户的最终志愿草案

验收：

- 运营能在 3 分钟内看懂某个推荐为什么出现。
- 能人工排除明显不适合的专业组。

## 数据质量检查清单

每次更新官方数据后，都要检查：

- 总记录数是否异常变化
- 院校数量是否异常变化
- 专业组数量是否异常变化
- 科类分布是否正常
- 本科批和专科批数量是否正常
- 计算机、临床医学、法学、师范等热门专业是否能查到样例
- 专业组代码是否能和历史推荐里的“专业组xxx”匹配
- 学费、计划人数、位次字段是否存在大量空值

- 无专业建议时，不显示空模块，不影响原候选卡片。

### 口径验收

- 不出现“保证录取”“录取概率百分比”等不严谨表达。
- 不新增数据库外学校和专业。
- 明确提示正式填报以四川省教育考试院、阳光高考平台和高校招生章程为准。

## 最小上线顺序

1. 本地确认官方 Excel 已提取为 NDJSON。
2. 执行 Prisma 生成。
3. 创建或同步 `gaokao_major_plans` 表。
4. 导入 `data/gaokao/sichuan-2026-major-plans.ndjson`。
5. 跑推荐逻辑测试。

8. 服务器执行数据库同步和专业计划导入。

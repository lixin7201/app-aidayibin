# AI 写真真实生成流程问题复盘与优化需求

日期：2026-05-17

## 一、背景

线上反馈：用户在 App 内上传照片后点击“生成”，页面提示“系统开小差了，请稍后再试”。

本次排查对照了当前版本和旧部署包 `app-aidayibin-deploy-20260516-205433.tar.gz`，并本地模拟了 App WebView 用户流程：

1. 通过 QFH5 用户信息建立 H5 登录态。
2. 上传 1 张图片。
3. 提交 AI 写真生成任务。
4. 拉取生成记录并同步任务状态。

## 二、已确认根因

### 2.1 直接原因

当前版本在写真和算命生成服务中新增了 MySQL 顾问锁，SQL 写法为：

```sql
SELECT GET_LOCK(..., 5) as lock
```

`lock` 在 MySQL 中会触发语法错误。这个异常不是业务 `AppError`，因此接口统一兜底返回：

```text
系统开小差了，请稍后再试
```

影响接口：

- `POST /ai/api/generations`
- `POST /ai/api/fortune/generations`

### 2.2 已修复方案

1. 抽出统一 MySQL 顾问锁工具。
2. 将返回别名改为安全字段：

```sql
SELECT GET_LOCK(..., 5) as lock_status
```

3. 兼容 MySQL/Prisma 返回 `number`、`bigint` 或字符串形式的 `1`。
4. 写真和算命生成服务都改为复用同一工具。
5. 增加单元测试覆盖锁别名和释放逻辑。

## 三、已完成验证

本地已验证：

1. MySQL 真实探针：`GET_LOCK(... ) as lock_status` 可正常获取和释放锁。
2. QFH5 mock 登录成功。
3. 上传接口成功返回图片 URL。
4. 写真生成提交接口返回 `200 OK` 和 `task_id`。
5. 拉取生成记录后任务可同步为 `succeeded`。

## 四、仍需优化的问题

### 4.1 首屏登录态闪烁

现象：

App 内首次打开页面时，QFH5 登录交换需要一小段时间。交换完成前，右上角会短暂显示：

```text
登录后体验
```

风险：

用户会误以为没有识别到 App 登录态。

需求：

1. 增加“正在同步登录”状态。
2. 登录交换完成前不要显示“登录后体验”。
3. 超过 3 秒未完成时，再提示用户重新进入 App 或检查登录状态。

建议文案：

```text
正在同步登录
```

超过 3 秒：

```text
登录信息同步较慢，请稍候
```

失败：

```text
未获取到 App 登录信息，请重新进入页面
```

### 4.2 生成提交后的可见反馈不足

现状：

提交成功后只更新顶部提示：

```text
提交成功，图片正在生成中。
```

如果用户没有展开“生成记录”，看不到任务卡片和持续状态。

需求：

1. 提交成功后立即显示一个当前任务状态条。
2. 状态条不依赖“生成记录”是否展开。
3. 状态应分阶段展示：
   - 上传中
   - 提交生成中
   - 排队中
   - 生成中
   - 生成成功
   - 生成失败

建议文案：

```text
照片已提交，正在生成写真
```

超过 10 秒：

```text
生成通常需要几十秒，请保持页面打开
```

超过 60 秒：

```text
图片仍在生成中，你可以稍后回到生成记录查看
```

### 4.3 错误提示缺少分层

现状：

非业务异常会统一显示“系统开小差了”，用户不知道是上传失败、提交失败、额度问题、AI 服务问题还是网络问题。

需求：

前端按照错误来源展示更准确文案：

| 场景 | 建议文案 |
| --- | --- |
| 上传接口失败 | 照片上传失败，请检查网络后重试 |
| 存储未配置/上传存储失败 | 图片存储暂时不可用，请稍后再试 |
| 生成接口参数错误 | 照片或模板信息异常，请重新选择后提交 |
| AI 服务提交失败 | AI 服务繁忙，请稍后重试 |
| 数据库/系统异常 | 系统暂时不可用，请稍后再试 |
| 已有任务生成中 | 你已有一张图片正在生成，请稍后查看 |

服务端要求：

1. 对 Zod 参数校验错误返回明确 `INVALID_REQUEST`，不要走 `UNKNOWN_ERROR`。
2. 对 Prisma 常见错误记录服务端日志，但前端只展示安全文案。
3. 关键接口日志必须包含 request id、user id、接口路径、错误 code。

### 4.4 生成记录刷新不够主动

现状：

历史记录默认折叠。提交成功后，如果历史没有展开，当前实现不一定拉取并展示最新任务卡。

需求：

1. 提交成功后即使历史折叠，也要保存当前任务摘要。
2. 页面每 3-5 秒轻量轮询当前任务，直到成功/失败。
3. 用户展开历史时，不应再看到旧状态。
4. 页面重新进入时，先同步 pending/processing 任务。

### 4.5 上传阶段缺少细粒度状态

现状：

上传和提交共用一句：

```text
正在上传照片并提交生成任务。
```

需求：

1. 上传开始时显示“正在上传照片”。
2. 多图上传时显示 `正在上传第 1/4 张`。
3. 上传完成后显示“照片已上传，正在提交生成”。
4. 提交成功后显示“已进入生成队列”。

### 4.6 线上排错能力不足

现状：

用户看到“系统开小差了”时，需要人工查看 PM2 日志才能知道真实错误。

需求：

1. 每次接口错误返回一个短 request id，例如 `ERR-20260517-xxxx`。
2. PM2 日志中输出同一个 request id。
3. 前端错误提示可附带：

```text
错误编号：ERR-xxxx
```

4. 管理后台后续可增加“最近错误日志”视图。

## 五、优先级建议

### P0：必须立即上线

1. 修复 MySQL 锁 SQL 别名导致的生成失败。
2. 保证写真和算命两条生成链路都可提交任务。

### P1：下一版优化

1. 登录同步状态，避免“登录后体验”闪烁。
2. 提交成功后显示当前任务状态条。
3. 上传/提交/生成分阶段提示。
4. 服务端 Zod 错误和常见系统错误分层返回。

### P2：后续增强

1. request id 全链路排错。
2. 当前任务自动轮询和历史记录主动刷新。
3. 管理后台错误日志视图。

## 六、验收清单

1. App 内打开 `/ai/photo`，右上角不再误闪“登录后体验”。
2. 上传 1 张 JPG/PNG/WEBP 后可以点击生成。
3. 点击生成后用户立即看到阶段反馈。
4. `POST /ai/api/generations` 返回 `200 OK` 和 `task_id`。
5. 生成中任务在页面有持续状态。
6. 成功后能看到结果图、保存、分享。
7. 异常时能看到可理解文案，而不是只有“系统开小差了”。
8. `/ai/fortune` 的生成提交同样通过。

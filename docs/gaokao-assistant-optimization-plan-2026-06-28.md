# 大宜宾高考填报 AI 助手优化方案

日期：2026-06-28  
项目：`/Users/lixin/AI code/app-aidayibin`  
入口：`/ai/gaokao`  
部署策略：先只改本地，测试没问题后再部署到 `/www/wwwroot/ai-gaokao-test`

## 1. 当前结论

当前高考助手不能直接进入开发新功能阶段，必须先把基础可用性修好。

核心问题不是单点 UI bug，而是 4 条链路同时没闭环：

- 登录展示链路：已有全局 App 登录桥，但高考页没有像 AI 写真/AI 算命那样实时消费登录结果并展示头像。
- 数据推荐链路：当前推荐把专业偏好当硬过滤条件，但四川 ZYFZ 数据多数是“院校专业组”，不是细专业名，导致有位次候选也被过滤成空。
- 报告展示链路：LLM 返回 Markdown，前端直接原样展示，所以 `#`、`**`、表格线、分隔线全部露出。
- 分享裂变链路：高考报告当前没有专用报告卡片图，分享仍是普通 H5 链接能力，视觉和传播性弱。

第一阶段目标：让用户填写基本信息后，一定能得到一份可信、可读、可分享、可复看的四川志愿初筛报告。

## 2. 已确认需求

### 2.1 生成次数

- 一个用户只能生成一次正式推荐，正式推荐包括聊天确认、冲稳保推荐、报告生成。
- 已生成过的用户再次进入，只能查看旧报告、分享旧报告，不能继续聊天生成新推荐。
- 如果资料填错，可以删除旧报告后重生成一次。
- 删除重生成机会仅允许一次。也就是“首次正式生成 + 一次纠错重生成机会”。
- 第二次报告再删除后，不允许继续生成，需要后台或人工重置。

### 2.2 用户信息填写

高考页要从纯聊天改成“表单先行 + 顾问补充”的模式。

必填字段：

- 姓名：用户手填，用于报告和分享卡片展示。
- 科目：下拉或单选，物理 / 历史。
- 分数：数字输入。
- 位次：数字输入。

参考四川填报系统补充字段：

- 首选科目：物理 / 历史，单选。
- 再选科目：化学 / 生物 / 思想政治 / 地理，多选。
- 分数不含加分。
- 后续可补：批次、城市偏好、专业偏好、是否接受调剂、是否接受民办/中外合作、学费上限。

### 2.3 推荐深度

- 用户填了分数和必要信息后，必须至少给出若干院校/院校专业组推荐，不能空报告。
- 能细到专业最好，但第一版要如实区分：
  - 数据能支持细专业时，展示专业。
  - 数据只支持院校专业组时，展示院校专业组，并提示“组内专业需以 2026 招生计划核对”。
- LLM 可以直白吐槽城市、专业就业、学校层次和现实取舍，但不能冒犯用户、不能绝对承诺录取。

### 2.4 报告和分享

- 报告页可以展示完整报告。
- 分享页也可以展示完整报告，但必须有明确 CTA，引导“我也想测一份志愿初筛”。
- CTA 点击回到大宜宾 App / H5 的高考助手入口。
- 用户从微信打开别人分享的报告页时，优先自动获取微信用户头像和昵称，参考 AI 写真 / AI 看手相现有微信登录链路。
- 微信身份获取不能作为查看公开报告的前置条件；OAuth 或头像获取失败时，报告仍可阅读，CTA 仍可点击。
- 分享卡片风格走“升学报告 / 录取通知书 / 金榜题名”的可信正式方向。
- 小程序卡片分享可以放到第二版，但需求文档中要一次性写清楚。
- 第一版优先做漂亮报告页 + 普通 H5 分享 + 一张默认报告卡片图。

四川省教育考试院官网在 2026-06-25 发布了 2026 年普通高考物理类、历史类成绩分段统计表，因此模型继续说“2026 年高考尚未进行”是错误上下文导致的，不是用户输入问题。来源：四川省教育考试院历史类分段统计表，发布时间 2026-06-25 18:59:26；物理类分段统计表，发布时间 2026-06-25 18:58:11。

这意味着第一版推荐必须围绕“位次 + 首选科目 + 院校专业组”建立，不能只按专业关键词硬筛。

## 4. 当前代码问题定位

### 4.1 登录和头像

现状：

- `src/features/auth/app-token-login.tsx` 全局监听 `QFH5.getUserInfo`，成功后派发 `aidayibin:auth-ready`。
- `src/app/gaokao/page.tsx` 服务端读取 `getStoredSessionFromCookies()` 并传入 `currentUser`。
- `src/features/gaokao/gaokao-app.tsx` 只用 `currentUser` 控制按钮禁用和提示，没有展示头像。
- `aidayibin:auth-ready` 事件只触发 `refreshReports()`，没有更新页面里的用户头像状态。

影响：

- App 第一次打开高考助手时，即使桥接登录成功，页面仍可能停留在“未登录”视觉状态。
- 右上角没有头像，和 AI 写真/AI 算命体验不一致。

修复方向：

- 高考页增加本地 `currentUser` state。
- 监听 `aidayibin:auth-ready` 后更新昵称和头像。
- 提供 `/api/auth/check` 返回用户信息，避免只知道 session 是否存在。
- 顶部按写真/算命风格展示头像、昵称、报告状态。

### 4.2 2026 时间上下文

现状：

- `src/features/gaokao/gaokao-llm.ts` 的 system prompt 没有当前日期、四川 2026 成绩已发布、2026 分段统计已发布等上下文。
- LLM 会按模型旧知识回答“2026 年高考尚未进行”。

修复方向：

- 在所有高考 LLM system prompt 中写死当前业务事实：
  - 当前业务日期按服务器当前日期或配置注入。
  - 四川 2026 高考已结束。
  - 四川 2026 成绩和物理/历史分段统计已于 2026-06-25 发布。
  - 2026 录取结果尚未产生，不能说有 2026 录取位次，只能用 2025/2024 历史投档和 2026 分段/招生计划辅助判断。
- 报告中必须区分“2026 成绩/位次已知”和“2026 录取结果未出”。

### 4.3 Markdown 垃圾符号

现状：

- `summary` 是 LLM 自由文本。
- 前端用 `whitespace-pre-wrap` 原样输出。
- 所以 Markdown 标题、粗体、表格线、分隔线全部变成用户看到的垃圾符号。

修复方向：

- 不再让 LLM 输出 Markdown 报告。
- 报告数据改为结构化 JSON：
  - `headline`
  - `studentSnapshot`
  - `strategySummary`
  - `recommendationBuckets`
  - `riskNotes`
  - `nextActions`
  - `disclaimer`
- 前端用组件渲染，不展示原始 Markdown。
- 为兼容旧报告，增加一个极小的 Markdown 清洗函数，只用于历史旧文本兜底。

### 4.4 推荐为空

现状：

- `buildGaokaoRecommendations` 先按位次/分数找记录。
- `filterByPreference` 再用 `preferredMajors` 对 `school_name + major_name` 硬过滤。
- 四川 ZYFZ CSV 中 `major_name` 多是 `专业组101（选科：不限）` 这种院校专业组名，不含“计算机”“物理”等具体专业关键词。
- 示例：物理类位次 12000，冲稳保位次范围内本来有候选，但按“计算机/物理”过滤后为 0。

修复方向：

- 专业偏好不能做硬过滤，改为打分排序。
- 推荐必须分层兜底：
  - 第一层：位次范围 + 科类 + 批次 + 再选科匹配。
  - 第二层：专业/城市/学校层级偏好加分排序。
  - 第三层：若专业关键词无法命中，仍展示位次匹配的院校专业组，并提示“专业组内具体专业需核对招生计划”。
  - 第四层：若某一档为空，自动放宽该档 rank 区间，但必须标注“已放宽筛选范围”。
- 报告中禁止出现“数据库没给出现成菜名，所以我编几个学校”这种话。
- LLM 不允许新增推荐学校，只能解释后端已给出的候选。

### 4.5 张雪峰 agent 完整性

现状判断：

- 当前代码并没有完整调用一个独立的 `xuefeng-agent` 推理链。
- 现在更像是：
  - 导入了四川 ZYFZ CSV。
  - 自写了位次范围筛选。
  - 让 LLM 做解释和报告。
- 原始 `admission_clean.db` 本地查询四川为 0，不能作为四川推荐主数据源。
- 已只读检查原项目 `https://github.com/ziqihe10-droid/xuefeng-agent.git`，默认分支 `master`，当前 HEAD 为 `9fbf7e79895b92b23d8e1c04c57d02a5938bdb8d`。
- 原项目当前 `admission_clean.db.gz` 解压后约 260,884 条记录，省份覆盖不含四川，四川记录数为 0。
- 原项目采用 AGPLv3，并在 LICENSE/README 中明确“网络服务也需开源”和“闭源商用需单独商业授权”。因此第一版不能直接把其代码或数据库原样并入闭源业务，除非先处理授权。

优化方向：

- 产品对外文案不出现“张雪峰”或“雪峰 Agent”。
- 技术实现上借鉴原项目的方法论和交互结构：位次优先、关键词专业筛选、关键词筛不到时自动放宽、DB/联网来源标注、自定义数据优先、多轮追问。
- 不直接复制原项目代码；如后续确需原样使用其代码/数据库/话术，需先确认 AGPL 开源影响或取得商业授权。
- 第一版文案可表述为“基于四川官方投档数据、招生计划和 AI 顾问分析”。
- 如果要细专业推荐，需要补充数据：
  - 2026 在川招生计划中的专业组内专业清单。
  - 专业组与专业方向映射。
  - 学科/专业就业标签库。
  - 学校城市、省份、层级、公民办、中外合作等标签。

原项目可迁移的思路：

- 关键词命中时优先给专业相关推荐。
- 关键词筛空时自动退回“省份 + 位次/分数”推荐，避免空报告。
- 有自定义数据时优先使用用户数据，并用 2024/2025 两年位次做综合排序。
- 每条推荐标注数据来源，避免让用户误以为 AI 凭空判断。
- 追问重点包括家庭资源、就业目标、城市偏好、讨厌的学科、考研/考公/就业倾向。

## 5. 第一阶段实施方案

第一阶段只做本地修改，不部署服务器。

### 5.1 页面形态

把 `/ai/gaokao` 改成三段：

1. 顶部用户区
   - 标题：大宜宾高考填报 AI 助手。
   - 右上角展示头像和昵称。
   - 已生成过报告时显示“已生成报告，可查看/分享”。

2. 考生信息表单
   - 姓名输入。
   - 首选科目：物理 / 历史。
   - 再选科目：化学 / 生物 / 思想政治 / 地理。
   - 分数输入，提示“不含加分”。
   - 位次输入。
   - 专业偏好输入或快捷标签。
   - 城市偏好输入或快捷标签。
   - 是否接受调剂。
   - 是否接受民办 / 中外合作。

3. 顾问确认区
   - 表单填完后，AI 用一两句话确认信息和追问关键缺口。
   - 不再让用户在空聊天里从零开始说。
   - 用户确认后点击“生成我的志愿初筛报告”。

### 5.2 数据模型调整

建议在 `GaokaoProfile` 增加：

```ts
studentName: string | null;
firstChoiceSubject: "物理" | "历史" | null;
optionalSubjects: Array<"化学" | "生物" | "思想政治" | "地理">;
```

保留原有：

```ts
subjectType: "物理类" | "历史类" | null;
score: number | null;
rank: number | null;
preferredMajors: string[];
preferredCities: string[];
acceptPrivate: boolean | null;
acceptSinoForeign: boolean | null;
acceptAdjustment: boolean | null;
```

报告表建议增加：

```prisma
report_card_image_url String? @db.Text
generation_round Int @default(1)
```

也可以先不加字段，用 `recommendations` JSON 保存卡片所需摘要，卡片图按需生成。第一版更推荐少改表。

### 5.3 生成次数控制

规则：

- 用户没有任何高考报告：可以聊天、可以生成。
- 用户有未删除报告：进入只读模式，只能查看和分享。
- 用户删除过 1 次高考报告且当前无报告：允许重生成一次。
- 用户删除过 2 次或已有第二次生成记录：不允许继续生成，提示“本账号已用完本年度 AI 初筛机会，如确需重置请联系大宜宾客服”。

实现方式：

- 查询 `gaokao_reports` 当前用户所有记录，包含软删除。
- `activeReports = deleted_at is null`
- `totalReports = all reports count`
- 允许生成条件：
  - `activeReports === 0 && totalReports < 2`
- 生成后立即保存报告，不再把“推荐结果未保存”留在前端状态里。
- 删除报告只做软删除，不删除历史计数。

### 5.4 推荐算法调整

第一版不用新增复杂模型，先把现有规则修到可用。

推荐输入：

- 科类。
- 分数。
- 位次。
- 首选科目。
- 再选科目。
- 专业偏好。
- 城市偏好。
- 是否接受民办 / 中外合作 / 调剂。

推荐过程：

1. 按科类、位次区间找候选。
2. 按选科要求过滤明显不符合项。
3. 不用专业偏好硬过滤，改为打分：
   - 专业组或专业名命中偏好：加分。
   - 学校/城市命中偏好：加分。
   - 学校层次更强：加分。
   - 中外合作/民办与用户不接受冲突：降权或排除。
   - 位次更贴近档位中心：加分。
4. 每档至少尽力给出 3 条。
5. 如果确实无法按偏好命中，展示“位次匹配候选”，并说明专业需二次核对。

参考原 `xuefeng-agent` 的兜底策略：

- 先带 `keyword` 查专业/学校相关候选。
- 如果关键词把结果筛空，自动去掉关键词，仅按省份、位次或分数重查。
- 若有位次优先用位次，没位次再按分数范围兜底。
- 对用户自定义数据或官方新增数据，优先按两年位次平均排序，再切冲稳保。

推荐分档：

- 冲：历史最低位次略高于用户位次，建议只放少量。
- 稳：历史最低位次接近用户位次，是主阵地。
- 保：历史最低位次低于用户位次，用于降低滑档风险。

具体阈值第一版可继续使用现有比例，但必须可集中配置，后续根据真实反馈调整。

### 5.5 细专业能力

第一版数据主要是院校专业组，无法稳定推荐到具体专业。

要做到“像张雪峰那样推荐专业”，需要补充：

- 2026 四川招生计划：院校专业组内包含哪些专业。已确认允许继续从四川官方招生计划页面/文件整理该数据。
- 每个专业的专业类标签：计算机、电子信息、医学、法学、师范、财经等。
- 专业就业/考研/体制内/城市适配标签。
- 学校城市、层次、优势学科标签。

### 5.6 报告渲染

报告不要再用 Markdown。

推荐结构：

```ts
type GaokaoReportContent = {
  title: string;
  student: {
    name: string;
    subjectType: string;
    optionalSubjects: string[];
    score: number;
    rank: number;
  };
  summary: string;
  buckets: {
    chong: RecommendationCard[];
    wen: RecommendationCard[];
    bao: RecommendationCard[];
  };
  strategy: string[];
  risks: string[];
  nextActions: string[];
};
```

页面渲染：

- 顶部先给结论：“按你当前位次，建议主攻稳档，冲档控制数量，保底要留够”。
- 冲稳保三栏卡片。
- 每条推荐有：
  - 学校 / 院校专业组。
  - 年份。
  - 分数。
  - 位次。
  - 位次差。
  - 风险标签。
  - 一句话理由。
- 底部有官方核对提醒。

LLM 只生成解释字段，不负责推荐名单。

- 主按钮：`我也想测一份志愿初筛`

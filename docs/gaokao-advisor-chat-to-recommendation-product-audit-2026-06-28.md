# 高考志愿 AI 顾问聊天影响推荐链路产品审查与优化建议

日期：2026-06-28
项目：`app-aidayibin`
范围：只审查和提出方案，不修改程序逻辑

## 1. 一句话结论

当前高考志愿助手已经有基础闭环：聊天采集画像、按四川历史投档数据做冲稳保、生成报告。但它还不像一个严肃的“智能顾问”，主要问题不是页面文案，而是产品链路里缺了两件关键能力：

- 用户在聊天里表达的真实意愿，没有稳定变成可执行的推荐约束。
- 点击“开始智能分析”后，系统没有展示分析、校验、取舍过程，结果出来太快，用户会觉得像模板筛选。

截图里的“我想离家远点、远离原生家庭、东北好一些”，当前系统很容易只存成 `notes`，不会变成“排除四川省内院校、偏好东北地区院校”的硬规则。所以后面继续出现很多四川大学，是符合当前代码行为的，但不符合用户对 AI 顾问的期待。

## 2. 重要概念先拆清楚

这里有一个容易混淆的点：

- `province: 四川`：当前系统里的省份主要指“考生生源地/高考省份”，也就是四川考生。
- `school_province: 四川`：这是学校所在地，才是用户说“不要四川学校、离家远点”时真正要影响的字段。

用户说“远离四川”，不应该理解为“不服务四川考生”。正确理解是：

- 仍然用四川考生的位次、批次线、历史投档和 2026 招生计划。
- 推荐学校所在地尽量排除四川，或者至少降低四川省内学校优先级。
- 如果因为分数、位次、招生计划导致符合条件的外省候选不足，必须明确告诉用户“外省符合条件不足”，不能悄悄塞四川学校假装满足。

## 3. 当前代码现状审查

### 3.1 聊天画像字段不够表达真实地域意愿

当前 `GaokaoProfile` 只有这些地域相关字段：

- `preferredCities`
- `rejectedCities`

代码位置：

- `src/features/gaokao/types.ts`
- `src/features/gaokao/gaokao-profile.ts`

问题：

- 没有 `preferredSchoolProvinces`。
- 没有 `rejectedSchoolProvinces`。
- 没有 `preferredRegions`，比如东北、华东、华南。
- 没有 `distancePreference`，比如离家近、离家远、省内、省外。
- 没有 `locationStrictness`，无法区分“坚决不要四川”和“可以少放一点四川”。

因此，“东北好一些”“离家远点”“远离四川”这种真实咨询话术，目前没有稳定字段承接。

### 3.2 关键词抽取范围太窄

当前 `cityKeywords` 只列了一些城市，例如成都、重庆、北京、上海、广州、深圳、杭州、南京、武汉、西安。

问题：

- 没有哈尔滨、长春、沈阳、大连等东北核心城市。
- 没有四川、北京、辽宁、吉林、黑龙江等学校所在地省份。
- 没有东北、华北、华东、华南、西南、西北等区域。
- “远离四川”“省外”“离家远”不是城市词，当前无法进入 `preferredCities/rejectedCities`。

### 3.3 外部顾问结果也会被当前 Schema 限制

即使外部 AI 顾问能理解用户想“远离四川、偏好东北”，当前 `profilePatch` 也只能保留既有字段：

- `preferredMajors`
- `rejectedMajors`
- `preferredCities`
- `rejectedCities`
- `riskPreference`
- `tuitionLimit`
- `acceptPrivate`
- `acceptSinoForeign`
- `acceptAdjustment`
- `notes`

也就是说，外部顾问如果返回 `rejectedSchoolProvinces: ["四川"]` 或 `preferredRegions: ["东北"]`，当前系统没有字段接收，最后大概率被丢掉。

### 3.4 主推荐层缺少学校所在地过滤

当前主推荐从 `gaokaoAdmissionRecord` 取历史投档记录，字段主要是：

- 学校名
- 专业/专业组名
- 年份
- 分数
- 位次
- 批次

这张表没有直接的 `school_province` 和 `school_city` 字段。`school_province/school_city` 主要存在于 `gaokao_major_plans`，也就是 2026 官方招生计划专业明细表。

结果是：

- 专业建议层可以比较好识别学校所在省市。
- 主推荐学校层对“不要四川学校”过滤不完整。
- 如果只靠学校名文本过滤，“四川大学”容易被排除，但“西南财经大学”“电子科技大学”“成都信息工程大学”等会漏掉或不稳定。

### 3.5 “开始智能分析”本质是一轮快速接口

前端点击后直接请求：

- `/api/gaokao/recommend`

后端流程大致是：

- 补全分数对应位次。
- 按位次或分数查历史投档候选。
- 按冲稳保分档。
- 附加专业建议。
- 生成结构化报告。
- 保存并返回。

这条链路是确定性数据库计算，所以很快是正常的。但当前没有展示：

- 正在解析用户意愿。
- 正在确认硬约束。
- 正在匹配历史位次。
- 正在排除不符合意愿的学校。
- 正在核对 2026 招生计划。
- 哪些条件被满足，哪些条件冲突。

所以用户看到的是“瞬间出结果”，心理感受会变成“它根本没思考”。

### 3.6 报告摘要不是大模型深度生成

当前 `generateGaokaoReportSummary` 实际返回的是本地结构化报告内容，不是深度 AI 推理报告。

好处：

- 快。
- 稳。
- 不容易乱编学校。

问题：

- 缺少“顾问分析过程”。
- 缺少“为什么这些学校留下，那些学校被排除”。
- 缺少“用户意愿和分数现实发生冲突时的取舍说明”。

## 4. 产品风险判断

这是一个严肃商业产品，不能把它做成“看起来会聊天，实际不听用户话”的工具。当前最大风险是：

### 风险 1：用户明确表达限制，但结果违反限制

例子：

用户说：

> 我想离四川远点，东北好一些。

结果仍大量推荐四川省内学校。

用户感受：

- AI 没听懂。
- 顾问聊天是装饰。
- 推荐结果不可信。

### 风险 2：系统为了凑满冲稳保数量，牺牲用户硬约束

如果用户说“坚决不读民办”“坚决不去四川”，系统不能为了凑够 5/8/5 条候选就把这些学校放进去。

正确做法：

- 硬约束优先。
- 符合条件不足时，少出结果也可以。
- 必须提示“符合你限制的候选不足，需要你放宽某项限制后再补充”。

### 风险 3：智能感靠假等待，而不是靠可解释

不建议为了显得聪明而故意 sleep 3 秒。

更合理的做法是：

- 显示真实分析步骤。
- 展示系统实际完成了哪些校验。
- 快也可以，但要让用户看到“快是因为数据库已结构化”，不是“随便吐结果”。

## 5. 推荐优化目标

目标不是把推荐交给大模型自由发挥。严肃志愿产品应该采用：

```text
用户聊天
  -> 结构化画像和约束
  -> 用户确认
  -> 确定性数据库筛选
  -> 硬约束过滤
  -> 软偏好排序
  -> 证据校验
  -> AI/本地话术解释
  -> 报告落库
```

推荐名单必须由历史数据、2026 招生计划、用户分数位次共同决定。AI 顾问负责理解、追问、解释和取舍，不直接编学校名单。

## 6. 详细优化方案

### 6.1 增加“学校所在地偏好”结构化字段

建议在 `GaokaoProfile` 增加：

```ts
locationPreference: {
  preferredRegions: string[];
  rejectedRegions: string[];
  preferredSchoolProvinces: string[];
  rejectedSchoolProvinces: string[];
  preferredSchoolCities: string[];
  rejectedSchoolCities: string[];
  distancePreference: "near_home" | "far_from_home" | "province_outside" | null;
  strictness: "hard" | "soft";
}
```

为了少改代码，也可以第一期不做嵌套对象，直接新增平铺字段：

```ts
preferredRegions: string[];
rejectedRegions: string[];
preferredSchoolProvinces: string[];
rejectedSchoolProvinces: string[];
distancePreference: "near_home" | "far_from_home" | "province_outside" | null;
locationStrictness: "hard" | "soft";
```

推荐第一期用平铺字段，改动最小。

### 6.2 明确地域词典

需要新增一个小型地域词典，不要上复杂依赖。

第一期覆盖：

```ts
const regionProvinceMap = {
  东北: ["辽宁", "吉林", "黑龙江"],
  华北: ["北京", "天津", "河北", "山西", "内蒙古"],
  华东: ["上海", "江苏", "浙江", "安徽", "福建", "江西", "山东"],
  华中: ["河南", "湖北", "湖南"],
  华南: ["广东", "广西", "海南"],
  西南: ["重庆", "四川", "贵州", "云南", "西藏"],
  西北: ["陕西", "甘肃", "青海", "宁夏", "新疆"],
}
```

城市先补关键高考咨询高频城市：

- 东北：沈阳、大连、长春、哈尔滨。
- 华北：北京、天津、石家庄、太原。
- 华东：上海、南京、杭州、苏州、合肥、济南、青岛、厦门、福州。
- 华中：武汉、长沙、郑州。
- 华南：广州、深圳、珠海、佛山。
- 西南：重庆、成都、昆明、贵阳。
- 西北：西安、兰州、银川、乌鲁木齐。

### 6.3 把“自然语言意愿”转成硬/软约束

建议规则：

| 用户表达 | 结构化结果 | 约束级别 |
| --- | --- | --- |
| 不想留四川 / 不要四川 / 远离四川 / 离家远点 | `rejectedSchoolProvinces: ["四川"]` 或 `distancePreference: province_outside` | 默认 hard |
| 东北好一些 / 想去东北 | `preferredRegions: ["东北"]`，展开为辽宁、吉林、黑龙江加分 | soft |
| 必须东北 / 只看东北 | `preferredRegions: ["东北"]`，`locationStrictness: hard` | hard |
| 成都重庆优先 | `preferredSchoolCities: ["成都", "重庆"]` | soft |
| 不去成都 / 不要重庆 | `rejectedSchoolCities` | hard |

特别说明：

- “远离原生家庭”是强烈个人诉求，不应该只当备注。至少应转成“省外优先/排除四川省内院校”并让用户确认。
- 如果表达里有“可以”“尽量”“优先”，默认 soft。
- 如果表达里有“坚决”“不要”“不想”“远离”，默认 hard。

### 6.4 顾问聊天后增加“意愿确认卡”

在“开始智能分析”前，页面应明确展示：

```text
已确认硬信息：
四川历史类，500 分，全省位次 46498。

已理解的偏好：
专业方向：历史、地理、思想政治相关。
地域偏好：东北优先。
明确排除：四川省内院校。
录取策略：均衡。

还未确认：
民办是否接受；中外合作是否接受；学费预算。
```

如果用户聊天里出现“远离四川”但系统没有提取成“明确排除：四川省内院校”，就不应该让用户误以为已经理解。

交互建议：

- “开始智能分析”按钮文案改为 `按以上条件开始分析`。
- 如果存在 notes 但没有被结构化的强意愿，按钮上方提示：`有一条偏好还没确认：是否排除四川省内院校？`
- 用户点确认后再推荐。

### 6.5 主推荐层必须使用学校所在地

第一期不要大改数据库结构，优先复用已有 `gaokao_major_plans` 表。

推荐做法：

1. 先按历史位次从 `gaokaoAdmissionRecord` 取候选。
2. 根据候选学校名和科类，到 `gaokaoMajorPlan` 查学校所在地。
3. 对候选补上 `schoolProvince`、`schoolCity`。
4. 再执行地域硬过滤。
5. 过滤后数量不足时，放宽位次范围继续查，但不能放宽硬约束。
6. 仍不足时，返回不足提示，不要用四川学校硬凑。

这样避免第一期做数据库迁移，也能解决大多数“学校所在地”问题。

### 6.6 硬约束和软偏好分开处理

推荐规则：

硬约束直接排除：

- 明确不去的学校所在地省份/城市。
- 明确不读的专业。
- 不接受民办。
- 不接受中外合作。
- 学费超过预算。
- 选科不符合。

软偏好只加权：

- 东北优先。
- 城市更大优先。
- 公办优先但不是绝对。
- 专业更匹配优先。
- 学校层次更高优先。

这点很重要。用户说“东北好一些”不等于只能东北；但用户说“远离四川”就不能继续大量推荐四川。

### 6.7 推荐结果展示“为什么符合你的条件”

每个推荐项建议展示一行：

```text
符合条件：省外院校 / 东北地区 / 历史类可报 / 位次接近 / 公办
风险提醒：2025 历史最低位次比你靠前 1800 名，适合冲，不适合当稳
```

如果某个结果违反软偏好，也要说清楚：

```text
地域不完全匹配：该校不在东北，但位次和专业匹配度较高，因此放入稳档候选。
```

如果违反硬偏好，默认不展示。

### 6.8 增加真实“分析过程”状态

点击后不要直接空白等待，也不要假装模型长时间思考。建议展示一个真实进度面板：

```text
正在分析：
1. 已解析考生画像：历史类，500 分，位次 46498
2. 已确认地域限制：排除四川省内院校，东北优先
3. 正在匹配四川历史投档位次
4. 正在核对 2026 招生计划和学校所在地
5. 正在按冲稳保分档
6. 正在生成风险说明和下一步建议
```

实现建议：

- 前端可先用本地状态展示步骤。
- 后端返回 `analysisTrace`，包含真实执行结果。
- 如果实际计算很快，也至少在结果页顶部保留“分析过程摘要”，而不是只靠等待动画。

`analysisTrace` 示例：

```ts
type GaokaoAnalysisTrace = {
  extractedConstraints: string[];
  hardFiltersApplied: string[];
  softPreferencesApplied: string[];
  candidatePool: {
    initial: number;
    afterHardFilters: number;
    afterLocationFilters: number;
  };
  dataSources: string[];
  warnings: string[];
}
```

### 6.9 结果不足时要诚实，不要乱推荐

严肃产品必须允许“不足”。

例如用户 500 分历史类，要求：

- 不去四川。
- 只看东北。
- 只看公办。
- 不接受调剂风险。

如果符合条件很少，系统应该说：

```text
按你当前限制，符合位次和地域条件的候选不足。为了不乱推荐，我先不强行凑满冲稳保。

你可以选择：
1. 保留“排除四川”，但把“东北优先”放宽为“省外优先”。
2. 保留东北优先，但允许少量四川省外的其他北方城市。
3. 先看稳保，再单独看冲刺项。
```

不要为了页面好看固定展示 18 条。

### 6.10 顾问回复要能主动追问关键矛盾

用户说：

> 我没有方向，你直接帮我做决定吧。

顾问不能直接拍脑袋决定。它应该追问或给可确认的默认策略：

```text
我可以先按“省外优先、东北加权、录取稳妥、专业不过窄”给你做第一版。但有三个风险需要你确认：

1. 是否明确排除四川省内院校？
2. 是否接受民办或高学费中外合作？
3. 专业上是否有坚决不读的方向？
```

如果用户已经明确说“远离四川”，系统应该主动复述：

```text
我先按“排除四川省内院校、省外优先、东北加权”理解。后面推荐如果外省候选不足，我会先提示你，不会偷偷把四川学校塞回来。
```

### 6.11 增加“顾问意图事件”，保证聊天可审计地影响结果

聊天不能只沉淀成一段 `notes`。每一轮用户输入，都应该先抽取成一组可审计的 `AdvisorIntentEvent`，再合并到 `GaokaoProfile` 和推荐约束里。

建议结构：

```ts
type AdvisorIntentEvent = {
  rawText: string;
  category:
    | "hard_fact"
    | "hard_constraint"
    | "soft_preference"
    | "emotional_need"
    | "uncertain_request"
    | "contradiction"
    | "unsafe_or_impossible"
    | "privacy_or_security"
    | "report_note";
  field:
    | "subjectType"
    | "score"
    | "rank"
    | "preferredMajors"
    | "rejectedMajors"
    | "preferredRegions"
    | "rejectedRegions"
    | "preferredSchoolProvinces"
    | "rejectedSchoolProvinces"
    | "preferredSchoolCities"
    | "rejectedSchoolCities"
    | "tuitionLimit"
    | "acceptPrivate"
    | "acceptSinoForeign"
    | "acceptAdjustment"
    | "riskPreference"
    | "notes";
  normalizedValue: unknown;
  confidence: "high" | "medium" | "low";
  strictness: "hard" | "soft" | "confirm";
  recommendationEffect:
    | "hard_filter"
    | "soft_boost"
    | "soft_penalty"
    | "follow_up_required"
    | "report_only"
    | "refuse_or_warn";
  needsUserConfirmation: boolean;
}
```

合并规则：

- 硬事实优先：科类、分数、位次以用户明确填写或最新明确表达为准，外部顾问不能覆盖成另一个分数或科类。
- 最新明确表达优先：用户后面说“刚才说错了，不排除四川了”，需要更新约束，但要在确认卡显示“已取消排除四川”。
- 不能从沉默推断：用户没说不接受民办，就不能默认排除民办；只能标记“未确认”。
- 情绪表达要转成可执行偏好：例如“想远离原生家庭”不是普通备注，应转成“省外优先/排除四川省内院校待确认”。
- 矛盾表达不能硬合并：例如“只看东北，又想成都重庆优先”，必须追问优先级。
- 不可能或高风险要求不能执行：例如“保证录取”“编一个能上的学校”“不要看位次”，必须拒绝承诺并解释边界。
- 每条推荐都必须能反查命中了哪些意图事件，至少在调试日志或报告摘要里保留。

### 6.12 顾问回复要同时满足情感需求和决策需求

顾问回复不要只做冷冰冰的字段提取，也不要只安慰不落地。每次回复建议固定三层：

```text
1. 接住情绪或复述意图：我理解你想离家远一点，这不是小事。
2. 转成可执行规则：我会先按“排除四川省内院校、省外优先、东北加权”理解。
3. 告知边界和下一步：如果符合位次的省外候选不足，我会先提示你，不会乱塞结果。
```

对不同输入的回复原则：

- 用户情绪强：先承接，再结构化，不嘲讽、不说教。
- 用户要求直接决定：可以给默认策略，但必须让用户确认关键风险。
- 用户要求保证录取：拒绝保证，只能给风险分档和核对依据。
- 用户条件矛盾：不假装理解，先问优先级。
- 用户明显焦虑：减少吓人话术，用“可以先做第一轮初筛”降低决策压力。
- 用户家庭冲突：尊重用户意愿，但不鼓励冲动决策，把它转成地域、预算、沟通成本等可讨论条件。

### 6.14 百问百答库怎样节省 token

要避免把“智能”理解成每句话都让大模型长篇推理。省 token 的关键是把稳定规则前置。

推荐实现：

- 本地规则库保存 `caseKey -> intentEvents -> responseTemplate`。
- 用户输入先走本地规则匹配，例如“远离四川”直接命中 `location.reject_sichuan`。
- LLM 只在三类情况调用：多意图混杂、情绪表达需要更自然回应、条件矛盾需要组织解释。
- LLM 输入只给压缩后的 JSON，不给全量聊天、不传 5 万行招生数据、不传完整百问百答库。
- 报告生成时只传候选摘要，不传所有候选和长备注。
- 没有 LLM 配置时，模板也能完成确认和推荐约束落地。

建议给 LLM 的小上下文：

```json
{
  "userMessage": "我想离家远点，东北好一些",
  "profileSummary": {
    "subjectType": "历史类",
    "score": 500,
    "rank": 46498
  },
  "matchedCases": ["location.reject_sichuan", "location.prefer_northeast", "emotion.family_distance"],
  "intentEvents": [
    {
      "field": "rejectedSchoolProvinces",
      "normalizedValue": ["四川"],
      "strictness": "confirm",
      "recommendationEffect": "hard_filter"
    },
    {
      "field": "preferredRegions",
      "normalizedValue": ["东北"],
      "strictness": "soft",
      "recommendationEffect": "soft_boost"
    }
  ],
  "replyGoal": "确认是否排除四川省内院校，并说明东北会优先但仍受位次约束"
}
```

这样每轮对话可以控制在很小的 token 内，同时推荐约束由结构化事件驱动，不依赖 LLM 是否临场发挥稳定。

### 6.15 百问百答库如何影响推荐，而不只是影响聊天

每条 `caseKey` 必须绑定推荐影响，不能只绑定回复文案。

示例：

| caseKey | 触发表达 | 推荐影响 | 页面呈现 |
| --- | --- | --- | --- |
| `location.reject_sichuan` | 远离四川、不想留四川 | 四川省内学校 hard filter | 确认卡显示“明确排除：四川省内院校” |
| `location.prefer_northeast` | 东北好一些、想去东北 | 东北 soft boost | 确认卡显示“地域偏好：东北优先” |
| `emotion.family_distance` | 远离原生家庭、离家远点 | 省外偏好待确认 | 顾问追问“是否排除四川省内院校” |
| `budget.low` | 家里没钱、别太贵 | 高学费 soft penalty，追问预算 | 确认卡显示“学费预算待确认” |
| `risk.no_repeat` | 不想复读 | safe strategy | 推荐保底数量和提醒增强 |
| `promise.guarantee` | 保证录取 | 不进入推荐，回复风险边界 | 页面不展示“保证”字样 |

验收标准：

- 如果一个用户表达只进入聊天气泡，不影响 `profile`、`intentEvents`、`analysisTrace` 或报告，必须判断它是否只是情绪安抚；如果不是，就算漏提取。
- 如果用户表达被识别为 hard filter，推荐主列表不得出现违反项。
- 如果用户表达被识别为 soft boost，推荐结果可以不全满足，但必须在解释里说清楚为什么。
- 如果用户表达被识别为 contradiction，不能直接生成结果，必须先让用户确认优先级。

### 6.16 需要沉淀成自动化测试的刁钻集合

不要只做人工验收。建议把百问百答库里最关键的 30 条做成自动化测试：

- `mergeGaokaoProfile` 或新建 `extractAdvisorIntentEvents` 的单元测试。
- `applyAdvisorResult` 的合并测试。
- `buildGaokaoRecommendations` 的硬过滤测试。
- `buildGaokaoReportContent` 的报告呈现测试。

第一批自动化测试必须覆盖：

- 用户说“远离四川”后，`rejectedSchoolProvinces` 生效。
- 用户说“东北好一些”后，`preferredRegions` 生效但不是硬过滤。
- 用户说“只看东北”后，非东北学校不出现在主结果。
- 用户说“成都重庆优先”后，成都重庆加权但不覆盖位次。
- 用户说“不要民办、不接受中外、预算 2 万”后，高学费/民办/中外被过滤。
- 用户说“保证录取”后，系统拒绝承诺，不写入虚假录取概率。
- 用户同时说“只看东北”和“成都优先”后，必须返回待确认矛盾，不直接推荐。
- 用户表达强烈情绪时，顾问回复先承接，但仍把可执行偏好结构化。

## 7. 分期执行建议

### 第一期：最小可信修复

目标：让聊天意愿真的影响推荐，不再出现“说了远离四川，结果仍大量四川”的明显违和。

修改范围：

- `src/features/gaokao/types.ts`
- `src/features/gaokao/gaokao-schemas.ts`
- `src/features/gaokao/gaokao-profile.ts`
- `src/features/gaokao/gaokao-service.ts`
- `src/features/gaokao/gaokao-repository.ts`
- `src/features/gaokao/gaokao-report.ts`
- `src/features/gaokao/gaokao-app.tsx`
- 相关测试文件

具体动作：

- 增加学校所在地偏好字段。
- 增加地域词典和“省外/东北/远离四川”解析。
- 外部顾问 `profilePatch` 允许返回地域偏好字段。
- 推荐候选通过 `gaokao_major_plans` 补学校所在地。
- 硬过滤排除用户明确拒绝的学校所在地。
- 报告“顾问确认参考”显示地域硬约束和软偏好。
- 增加至少 5 个测试用例。

不做：

- 不重做整套 UI。
- 不引入新依赖。
- 不让大模型决定学校名单。
- 不做数据库大迁移，除非后续性能证明需要。

### 第二期：智能分析过程可见

目标：让用户感到系统确实理解、筛选、校验过，而不是瞬间吐模板。

具体动作：

- 新增分析步骤状态。
- 后端返回 `analysisTrace`。
- 结果页展示“本次分析依据”。
- 候选卡片展示“符合条件”和“风险提醒”。
- 结果不足时展示放宽建议。

### 第三期：顾问质量提升

目标：让聊天更像专业顾问，而不是关键词收集器。

具体动作：

- 顾问每轮回复必须复述已理解的硬约束。
- 强意愿没有结构化时主动确认。
- 对矛盾条件主动提醒，比如“既要远离四川，又只接受低风险保底，候选会变少”。
- 报告里增加“取舍建议”，说明为什么某些偏好没有完全满足。

### 第四期：数据审计和运营后台

目标：上线后可追责、可复盘。

具体动作：

- 保存每次生成的结构化画像快照。
- 保存硬过滤数量、软偏好排序依据。
- 后台可查看“用户说了什么、系统理解成什么、结果是否满足”。
- 运营可标记明显异常案例，作为后续测试用例。

## 8. 必须补的验收用例

### 用例 1：远离四川

输入：

```text
历史类 500 分，位次 46498。我接受，我就想离家远点，远离原生家庭。东北好一些吧。
```

期望：

- 画像显示：排除四川省内院校，东北优先。
- 推荐主列表不应大量出现四川省内院校。
- 如果外省候选不足，应明确提示，不强行塞四川学校。

### 用例 2：只看东北

输入：

```text
物理类 520 分，位次 8 万，只看东北，其他地方先不考虑。
```

期望：

- `preferredRegions` 包含东北。
- `locationStrictness` 为 hard。
- 结果仅展示辽宁、吉林、黑龙江相关学校；不足则提示不足。

### 用例 3：东北优先但不绝对

输入：

```text
物理类 560 分，位次 5 万，东北优先，也可以看看其他省外城市。
```

期望：

- 东北学校排序加权。
- 可出现其他省外学校。
- 四川省内学校不应排在前面，除非用户明确允许。

### 用例 4：成都重庆优先

输入：

```text
物理类 590 分，位次 3 万，成都重庆优先，计算机方向，不接受民办。
```

期望：

- 成都、重庆加权。
- 民办硬排除。
- 推荐理由说明城市和专业匹配。

### 用例 5：条件过窄

输入：

```text
历史类 480 分，只看东北公办本科，不接受民办，不接受中外合作。
```

期望：

- 不乱推荐。
- 如果候选不足，给出放宽建议。
- 不为了凑数加入明显不符合条件的学校。

## 9. 验收标准

产品验收：

- 用户聊天里的强意愿，必须出现在确认卡和报告中。
- 推荐结果能解释每条为何符合或不完全符合用户条件。
- 用户明确排除的内容，不应出现在推荐主结果中。
- 候选不足时，宁可少推荐，也不乱推荐。
- 点击分析后有可见的分析过程或分析依据。

技术验收：

- `npm test` 通过。
- `npm run typecheck` 通过。
- `npm run lint` 通过。
- 新增地域偏好测试覆盖“远离四川、东北优先、只看东北、条件不足”。
- 推荐名单仍然只来自历史投档和 2026 招生计划，不由 LLM 编造。

数据验收：

- 每条推荐有历史位次或专业计划依据。
- 2026 录取结果未产生，不出现“2026 录取概率”“2026 已录取位次”等表述。
- 学校所在地来自结构化数据，不靠学校名称猜。

## 10. 建议优先级

优先级 P0：

- 地域意愿结构化。
- 排除四川省内院校这类硬约束生效。
- 开始分析前确认系统理解。
- 结果不足时不乱推荐。

优先级 P1：

- 分析过程可见。
- 结果卡片展示符合条件和风险提醒。
- 报告增加“本次顾问理解”和“取舍说明”。

优先级 P2：

- 后台审计。
- 运营异常案例沉淀。
- 更复杂的顾问策略。

## 11. 我建议的最小下一步

先不要大改 UI，也不要引入新模型服务。

最小可执行方案是：

1. 给画像补地域偏好字段。
2. 给聊天抽取补“远离四川、东北、省外”等规则。
3. 推荐时用 `gaokao_major_plans` 补学校所在地，再应用硬过滤。
4. 前端确认卡展示“系统理解到的硬约束和软偏好”。
5. 增加测试，确保“远离四川”不会再被当成普通备注。

这一步做完，用户会明显感觉：我说的话真的进入推荐了。

之后再做分析过程可见，让产品从“能用”进一步变成“可信、专业、有顾问感”。
